# Working with Wynxx Story Creator Backlogs

Use this skill to interact with backlogs and work items generated or managed by Wynxx.

## Important Concepts

### Work Item Hierarchy
```
Backlog
└── EPIC
    └── FEATURE
        └── USER_STORY
            ├── TASK
            └── TEST_CASE
```

- **Backlog**: groups EPICs generated from a feature description
- **Work Item**: any item in the hierarchy (Epic, Feature, User Story, Task, Test Case)
- A backlog belongs to a **Project** (`projectId`)
- The `initialLevel` field of the backlog indicates from which level the generation started (e.g., `EPIC`)

### Backlog Status
- `PENDING` — queued, not yet started
- `PROCESSING` — generation in progress
- `SUCCESS` — completed successfully
- `ERROR` — failed during generation

### Project Types (`projectType`)
- `DEFAULT` — internal Wynxx project
- `AZURE_BOARDS` — integrated with Azure DevOps
- `JIRA` — integrated with Jira

## Flow: List and Inspect Backlogs

### 1. List available projects
```
mcp_wynxx_list_projects
  - optional parameters: page (default 1), limit (default 10), projectType
  - returns: id, name, description, projectType, generationConfig
```
> Note: there may be hundreds of projects. Use `page` to navigate. Filter by `projectType` if needed.

### 2. List backlogs
```
mcp_wynxx_list_backlogs
  - optional parameters: page, limit, projectId (to filter by project), status
  - returns: id, title, description, projectId, projectName, status, initialLevelName, personaName
```
> To find the correct backlog, filter by `projectId` of the relevant project.
> Backlogs with `ERROR` status have no work items.

### 3. Get full backlog with hierarchy
```
mcp_wynxx_get_backlog
  - required parameter: backlogId (int)
  - returns: all work items in hierarchical structure (nested children)
```
> This is the main tool to view the generated content: Epics, Features, User Stories, etc.

### 4. Get details of a specific work item
```
mcp_wynxx_get_work_item
  - required parameter: workItemId (int)
  - returns: title, description, type, priority, businessValue, estimatedEffort, storyPoints,
             acceptanceCriteria, children (child IDs)
```
> Use when you need the full content of a specific item (e.g., acceptance criteria of a User Story).

## Flow: Analyze Backlog Coverage

When the user asks to verify whether all stories from an Epic/Feature have been implemented:

1. Call `mcp_wynxx_get_backlog` with the `backlogId` of the relevant backlog
2. Traverse the work item hierarchy to list all User Stories (type `USER_STORY`)
3. Compare with what has been implemented in the code
4. For details of a specific item (e.g., acceptance criteria), call `mcp_wynxx_get_work_item`

**Expected output example when analyzing coverage:**
```
Epic 4923 - Area Management
  ✅ US-4924: Create area (implemented in POST /areas)
  ✅ US-4925: List active areas (implemented in GET /areas)
  ❌ US-4930: Deactivate area — NOT implemented
```

## Tips and Best Practices

| Situation | What to do |
|-----------|------------|
| Too many projects returned | Filter by `projectType` or use pagination |
| Backlog with ERROR status | Do not try to access work items — check error message in the `errorMessage` field |
| Work item without `children` | The backlog may have been generated with `initialLevel: EPIC` without breakdown — check the project's `generationConfig.enableSplitting` |
| Need acceptance criteria | Use `mcp_wynxx_get_work_item` on the specific item (not always included in `get_backlog`) |
| IDs are integers | `backlogId` and `workItemId` are always `int`, not strings |
| AZURE_BOARDS project | The `externalId` field in the work item contains the Azure DevOps item ID after synchronization |

## Full Practical Example

**Scenario**: _"List the backlogs of the 'juar' project and show the User Stories of the first one"_

```
1. mcp_wynxx_list_projects → find project "juar" (ID: 394)
2. mcp_wynxx_list_backlogs with projectId: 394 → find backlog (e.g., ID: 542)
3. mcp_wynxx_get_backlog with backlogId: 542 → get full hierarchy
4. Filter work items with type == "USER_STORY" and present to the user
```
